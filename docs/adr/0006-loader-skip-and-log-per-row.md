# Loader skips and logs bad rows instead of failing the batch

The extraction loader treats each `logical_records[]` item as independently loadable: any row that fails semantic validation (open-ended range, inverted range, parity-mismatched range, unknown voie type or suffix, cross-quartier ilot, empty/unparseable `numeros_raw`, missing `ilot_numbers`) is dropped from the batch and reported in a structured `skipped[]` array, while the other rows still insert. Only **shape** errors (zod fails on `document_scope`, malformed JSON, missing required fields at the document level) abort with `400` and zero writes. The contract: a 200 response carries a `LoaderReport` with `inserted` counts per table and a `skipped` array of `{ reading_order_index, reason: SkipReason, detail }`; reasons are a typed enum so CLI grouping and future QA tooling can aggregate them. Pre-validation in TypeScript mirrors every schema constraint (`street_segments_range_order_ok`, `street_segments_parity_ok`, `segment_ilots_quartier_consistency`) so the actual D1 batch insert is always "trusted clean" and the constraints serve as defense in depth rather than the primary error path.

## Considered Options

- **Hard-fail on the first bad row.** Rejected: contradicts the "skip + log" doctrine already codified in `docs/EXTRACTION.md` for every documented no-go case. A single bad row would block hundreds of good rows on the same bobine until hand-corrected — the exact opposite of what the iterative LLM-extraction → manual-QA workflow needs. With the whole-bobine wipe (`docs/adr/0004-…`) the bobine is gone the moment the batch starts, so a hard-fail leaves the lookup table empty for that bobine until the JSON is fixed.
- **Configurable strict / lenient mode (`?strict=1` or `--strict`).** Rejected: surface area without payoff. The lenient mode is correct for the LLM-iteration workflow we have today; strict mode would only be useful for a later "production-quality publish" gate, which doesn't exist yet and can be added as a `--require-zero-skips` exit-code flag on the CLI without changing the route contract.
- **Append-only audit log table for skipped rows.** Rejected for v0: the JSON file in git is already the audit trail for what the LLM emitted, and the loader response carries every skip with its `reading_order_index` and reason. Adding a `loader_audit_log` table would persist transient data without a current consumer. Revisit when a QA dashboard needs historical skip aggregation across loads.
