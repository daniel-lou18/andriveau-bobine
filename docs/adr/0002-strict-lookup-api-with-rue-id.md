# Strict lookup API keyed by `rue_id`; disambiguation lives in the client

The address lookup endpoint receives an already-resolved `rue_id` (plus `n`, `n_rank`, `parity`) rather than a free-text rue string. A separate suggest endpoint backs an autocomplete on `rues.libelle_normalized` and returns `{ rue_id, type, libelle }` rows. The web client forces the user to pick a suggestion before submitting the main query, so the API never has to parse a rue or guess a `type` for a homonymic libellé (`Rue de Vaugirard` vs `Boulevard de Vaugirard`). This keeps the API contract small and aligned with the unique key `(type, libellé_normalized)` on `rues`: the disambiguation step is performed exactly once, at the moment the human can resolve it cheaply, and never replicated as a guess in the parser. It also avoids a second French-address parser inside the Worker that would compete with the extractor's parser and inevitably drift.

## Considered Options

- **Free-text lookup endpoint** that accepts a string like `"Bd Raspail 95"` or `"raspail 95"` and resolves server-side. Rejected: requires a parser symmetric to the extractor (two implementations of "parse a French address fragment"), forces an arbitrary policy when a libellé matches multiple `type`s (prefer `Rue`? return all?), and pushes a domain-truth decision into a default the user can't see.
- **Hybrid: strict primary endpoint plus a separate free-text fallback.** Rejected for now: doubles the API surface, doubles the parser surface, and the "primary" path already covers the prompt's stated default use case. Revisit only if a paste-from-elsewhere workflow becomes a real requirement.
- **Pass `(type, libelle_normalized)` instead of `rue_id`.** Rejected: human-readable in URLs but couples the contract to a string shape that's already implementation-detail (the normalization rule). `rue_id` is opaque, stable, and forces the client through the suggest endpoint, which is exactly the disambiguation pinch point we want.
