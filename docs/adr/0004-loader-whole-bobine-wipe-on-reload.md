# Loader replaces a bobine atomically; no per-record upsert

The extraction loader is keyed on `source_entries.bobine`: every load of `bobineN-extraction.json` first deletes the bobine's existing `source_entries` (cascading to `street_segments` and `segment_ilots` via `ON DELETE CASCADE` added in a follow-up migration) and then re-inserts the batch inside the same D1 transaction. The JSON file in `data/extracted-tables/` is the authoritative state for that bobine; once it changes — and it will change frequently while the LLM extraction is being tuned and `low_confidence` rows hand-corrected — the previously loaded rows are stale by definition. `rues` and `ilots` rows that become unreferenced are left in place as benign reference data: `rues` is intentionally shared across bobines, and `ilots.number` is a stable Paris fact regardless of whether the current corpus still cites it. The audit trail for "what did the data look like at time T" lives in git on the JSON file, not in D1.

## Considered Options

- **Per-record upsert keyed on `(bobine, page, reading_order_index)`.** Rejected: `reading_order_index` is **not stable** across LLM re-runs. A re-run that adds a previously missed sticky-îlot inheritance, catches a `NEANT` skip the first pass left in, or stitches a downward N° overflow into one logical row will shift every subsequent index on that page. Upsert by an unstable key produces duplicates and orphans.
- **Append-only versioned loads with a `load_batch_id` on `source_entries` and "latest wins" at query time.** Rejected: heavyweight for a project whose audit trail already lives in git on the JSON, forces every read path to filter on the latest batch, and bloats the table with historical rows the lookup must ignore. Revisit if the LLM extractor ever becomes the only source of truth (no human-checked JSON commits).
- **Cascading at the application level instead of the schema level** (loader issues `DELETE` in dependency order manually). Rejected: workable but every contributor would have to remember the order. `ON DELETE CASCADE` on `segment_ilots.segment_id` and `street_segments.source_entry_id` lets the schema enforce it and keeps the loader's wipe a single statement.
- **Garbage-collect orphan `rues` / `ilots` after each wipe.** Rejected: `rues` is bobine-independent by design (the unique key is `(type_id, libelle_normalized)`, not bobine), and deleting an `ilots` row that another bobine still references would violate the trigger or break that other bobine's lookup. Reference rows accumulate; that's the intended behavior.
